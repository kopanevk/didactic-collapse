# Дидактический коллапс в синтетических образовательных объяснениях

Репозиторий содержит материалы исследования «Дидактический коллапс в синтетических образовательных объяснениях: измерение, дефектно-бюджетное рециклирование и контрастивный анализ».

Исследование посвящено не деградации параметров модели, а качеству повторно используемого корпуса синтетических образовательных объяснений. В работе рассматривается ситуация, при которой модель создаёт набор учебных объяснений, затем часть этих объяснений отбирается и используется при формировании следующего поколения данных. Такой процесс обозначается как рециклирование синтетических объяснений.

## Что изучается

В исследовании вводится понятие **дидактического коллапса**.

Под дидактическим коллапсом понимается сохранение или распространение педагогически значимых дефектов при повторном использовании синтетических образовательных объяснений.

В отличие от классического `model collapse`, здесь не анализируется ухудшение самой модели. Параметры модели не изменяются. Предметом анализа становится состав повторно используемого корпуса объяснений: какие тексты переходят в следующее поколение и какие дефекты сохраняются вместе с ними.

## Основная идея

Синтетическое образовательное объяснение может выглядеть убедительно и даже содержать правильный финальный ответ, но при этом быть педагогически слабым: пропускать важные шаги, нарушать логику рассуждения, иметь слабую структуру или закреплять у учащегося ошибочный способ решения.

Поэтому в исследовании отдельно анализируются:

- точность финального ответа;
- педагогическая оценка объяснения;
- скрытые педагогические ошибки;
- ошибки разбора финального ответа;
- слабое рассуждение;
- слабая структура.

## Метод: дефектно-бюджетное рециклирование

В работе предлагается и проверяется **дефектно-бюджетное рециклирование** — `DBR` (`defect-budget recycling`).

DBR используется как способ отбора синтетических образовательных объяснений для следующего поколения данных. Его задача состоит не в том, чтобы удалить все несовершенные ответы, а в том, чтобы ограничить долю наиболее значимых дефектов в повторно используемом корпусе.

Процедура DBR включает следующие шаги:

1. для каждого объяснения вычисляются дефектные признаки;
2. рассчитывается интегральная тяжесть дефектов;
3. кандидаты упорядочиваются от менее дефектных к более дефектным;
4. объяснения последовательно добавляются в отобранный корпус;
5. при добавлении проверяется, не превышаются ли заданные дефектные бюджеты;
6. если исходные бюджеты не позволяют сохранить требуемый объём корпуса, ограничения ослабляются в фиксированном порядке;
7. итоговый набор и все нарушения бюджетов фиксируются для последующего анализа.

## Экспериментальная постановка

Эксперименты выполнялись на модели:

```text
qwen2.5:0.5b
```

Использовался режим:

```text
inference_recycling_only
```

Это означает, что модель применялась только для генерации новых объяснений по экспериментальному протоколу. Её параметры не обновлялись, а обучение на сгенерированных данных не проводилось.

В исследовании рассматривались три поколения данных:

```text
Gen0 → Gen1 → Gen2
```

Основными сравниваемыми ветвями были:

```text
обычное рециклирование синтетических объяснений
DBR
```

Дополнительно использовался контрастивный анализ: для одной и той же задачи генерировалось несколько объяснений, после чего сравнивались лучшие и худшие варианты.

## Основные результаты

На рассматриваемом объёме данных обычное рециклирование не показало устойчивого монотонного ухудшения всех ключевых метрик. Поэтому сильная форма дидактического коллапса не получила достаточной поддержки.

Однако дефекты не исчезали автоматически. В корпусах сохранялась ненулевая доля скрытых педагогических ошибок, слабого рассуждения, слабой структуры и ошибок разбора. Это соответствует слабой корпусной форме дидактического коллапса.

DBR показал наиболее устойчивый эффект в контроле ошибок разбора финального ответа: после отбора их доля снижалась до нуля при сохранении около 96–98% корпуса.

Влияние DBR на скрытые педагогические ошибки и педагогическую оценку оказалось менее устойчивым и требует дополнительной проверки на большем числе моделей, запусков и предметных областей.

## ZIP-архив воспроизводимости

В репозитории находится ZIP-архив:

```text
didactic_collapse_reproducibility_archive.zip
```

Это основной пакет воспроизводимости исследования. Он нужен для того, чтобы внешний читатель мог проверить, откуда взяты численные значения в статье и какие данные использовались при построении таблиц и рисунков.

Внутри архива находится папка:

```text
release_artifacts/
```

Структура архива:

```text
release_artifacts/
├── README.md
├── VALIDATION_REPORT.md
├── data/
│   ├── row_level/
│   ├── aggregated/
│   └── appendix/
├── scripts/
├── figures/
├── checksums/
│   └── SHA256SUMS.txt
└── manuscript_appendix/
    └── appendix_A.md
```

Содержимое архива:

- `README.md` — краткое описание структуры архива и назначения файлов;
- `VALIDATION_REPORT.md` — отчёт проверки численных значений статьи;
- `data/row_level/` — построчные результаты экспериментов на уровне отдельных объяснений;
- `data/aggregated/` — агрегированные таблицы, использованные для воспроизведения результатов статьи;
- `data/appendix/` — данные для Приложения A;
- `scripts/` — скрипты пересчёта, агрегации и проверки результатов;
- `figures/` — рисунки, использованные в рукописи и дополнительных материалах;
- `checksums/SHA256SUMS.txt` — контрольные суммы файлов архива;
- `manuscript_appendix/appendix_A.md` — версия Приложения A, соответствующая рукописи.

Файл `VALIDATION_REPORT.md` показывает, что заявленные в статье численные значения были сопоставлены с исходными таблицами. В финальной проверке отсутствующие значения и расхождения не выявлены:

```text
not found = 0
mismatch = 0
```

Файл `SHA256SUMS.txt` позволяет проверить целостность архива после скачивания или распаковки.

## Как проверить контрольные суммы

После распаковки архива перейдите в каталог, где находится папка `release_artifacts/`, и выполните:

```bash
sha256sum -c release_artifacts/checksums/SHA256SUMS.txt
```

На Windows можно использовать PowerShell или любую утилиту для проверки SHA256.
