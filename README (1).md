# Дидактический коллапс в синтетических образовательных объяснениях

Репозиторий содержит материалы исследования о том, как педагогические дефекты сохраняются при повторном использовании синтетических образовательных объяснений, созданных большой языковой моделью.

Работа посвящена не деградации параметров модели, а качеству повторно используемого корпуса объяснений. В исследовании рассматривается ситуация, когда модель генерирует учебные объяснения, затем часть этих объяснений отбирается и используется при формировании следующего поколения данных. Такой процесс называется рециклированием синтетических объяснений.

## Что изучается

В исследовании вводится понятие **дидактического коллапса**.

Под дидактическим коллапсом понимается сохранение или распространение педагогически значимых дефектов при повторном использовании синтетических образовательных объяснений.

В отличие от классического `model collapse`, здесь не анализируется ухудшение самой модели. Параметры модели не меняются. Анализируется то, какие объяснения переходят из одного поколения корпуса в следующее и какие дефекты вместе с ними сохраняются.

## Основная идея

Сгенерированное объяснение может выглядеть убедительно и даже содержать правильный финальный ответ, но при этом быть педагогически слабым:

- пропускать важные шаги рассуждения;
- нарушать логику решения;
- иметь слабую структуру;
- содержать скрытую педагогическую ошибку;
- закреплять у учащегося неверный способ рассуждения.

Поэтому в работе отдельно анализируются:

- точность финального ответа;
- педагогическая оценка объяснения;
- скрытые педагогические ошибки;
- ошибки разбора финального ответа;
- слабое рассуждение;
- слабая структура.

## Метод: DBR

В работе предлагается и проверяется **дефектно-бюджетное рециклирование** — `DBR` (`defect-budget recycling`).

DBR — это способ отбора объяснений для следующего поколения корпуса. Он не удаляет все несовершенные ответы, а ограничивает допустимую долю отдельных типов дефектов.

Идея метода:

1. для каждого объяснения вычисляются дефектные признаки;
2. рассчитывается интегральная тяжесть дефектов;
3. кандидаты упорядочиваются от менее дефектных к более дефектным;
4. объяснения последовательно добавляются в новый корпус;
5. при добавлении проверяется, не превышены ли дефектные бюджеты;
6. если строгие бюджеты не позволяют сохранить нужный объём корпуса, ограничения ослабляются в фиксированном порядке;
7. итоговый корпус и все нарушения бюджетов фиксируются для анализа.

## Экспериментальная постановка

Эксперименты выполнялись на модели:

```text
qwen2.5:0.5b
```

Использовался режим:

```text
inference_recycling_only
```

Это означает, что модель применялась только для генерации новых объяснений. Её параметры не обновлялись, а обучение на сгенерированных данных не проводилось.

В исследовании рассматривались три поколения данных:

```text
Gen0 → Gen1 → Gen2
```

Сравнивались две основные ветви:

```text
обычное рециклирование синтетических объяснений
DBR
```

Также использовался контрастивный анализ: для одной и той же задачи генерировалось несколько объяснений, после чего сравнивались лучшие и худшие варианты.

## Основные результаты

На рассматриваемом объёме данных обычное рециклирование не показало устойчивого монотонного ухудшения всех метрик. Поэтому сильная форма дидактического коллапса не получила достаточной поддержки.

Однако дефекты не исчезали автоматически. В корпусах сохранялась ненулевая доля скрытых педагогических ошибок, слабого рассуждения, слабой структуры и ошибок разбора. Это соответствует слабой корпусной форме дидактического коллапса.

DBR показал наиболее устойчивый эффект в контроле ошибок разбора финального ответа: после отбора их доля снижалась до нуля при сохранении около 96–98% корпуса.

Эффект DBR на скрытые педагогические ошибки и педагогическую оценку оказался менее устойчивым и требует дополнительной проверки на большем числе моделей, запусков и предметных областей.

## ZIP-архив воспроизводимости

В репозитории находится ZIP-архив:

```text
didactic_collapse_reproducibility_archive.zip
```

Это основной пакет воспроизводимости исследования. Он нужен для того, чтобы внешний читатель мог проверить, откуда взяты численные значения в статье и какие данные использовались при построении таблиц и рисунков.

Внутри архива находится папка:

```text
release_artifacts/
```

Она содержит:

- `README.md` — краткое описание структуры архива и назначения файлов;
- `VALIDATION_REPORT.md` — отчёт проверки численных значений статьи;
- `data/row_level/` — построчные результаты экспериментов на уровне отдельных объяснений;
- `data/aggregated/` — агрегированные таблицы, использованные для результатов статьи;
- `data/appendix/` — данные для Приложения A;
- `scripts/` — скрипты пересчёта, агрегации и проверки результатов;
- `figures/` — рисунки, использованные в рукописи и дополнительных материалах;
- `checksums/SHA256SUMS.txt` — контрольные суммы файлов архива;
- `manuscript_appendix/appendix_A.md` — версия Приложения A, соответствующая рукописи.

Файл `VALIDATION_REPORT.md` показывает, что заявленные в статье численные значения были сопоставлены с исходными таблицами. В финальной проверке отсутствующие значения и расхождения не выявлены:

```text
not found = 0
mismatch = 0
```

Файл `SHA256SUMS.txt` позволяет проверить целостность архива после скачивания или распаковки.

## Что находится в архиве

### `data/row_level/`

Построчные результаты экспериментов. Эти файлы содержат данные на уровне отдельных объяснений и используются как исходная база для пересчёта агрегированных результатов.

### `data/aggregated/`

Агрегированные таблицы, использованные для воспроизведения численных результатов статьи.

В частности, здесь находятся таблицы для проверки:

- механизма DBR;
- итоговых эффектов DBR на Gen2;
- контрастивного анализа;
- сопоставлений по сериям запусков.

### `data/appendix/`

Данные для Приложения A, включая размеры корпусов и состав экспериментальных серий.

### `scripts/`

Скрипты, использованные для агрегации, пересчёта и проверки результатов.

### `figures/`

Рисунки, использованные в рукописи и дополнительных материалах.

### `checksums/SHA256SUMS.txt`

Контрольные суммы файлов архива. Они позволяют проверить, что содержимое архива не было повреждено или изменено.

### `VALIDATION_REPORT.md`

Отчёт валидации численных значений. В нём указано, какие значения из статьи были найдены в исходных таблицах и совпали с пересчитанными результатами.

Финальный статус проверки:

```text
not found = 0
mismatch = 0
```

Это означает, что заявленные в статье численные значения были найдены в исходных таблицах, а расхождений не выявлено.

## Как проверить контрольные суммы

После распаковки архива можно проверить контрольные суммы.

На Linux/macOS:

```bash
cd release_artifacts
sha256sum -c checksums/SHA256SUMS.txt
```

На Windows можно использовать PowerShell или любую утилиту для проверки SHA256.
